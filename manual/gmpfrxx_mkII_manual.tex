\begin{abstract}
\noindent
\textbf{gmpfrxx\_mkII} provides value-semantic C++ classes for arbitrary-precision
integers, rationals, real floating-point numbers, and complex numbers, layered
on the GNU multiple-precision libraries GMP, MPFR, and GNU MPC. An
expression-template arithmetic layer removes most temporaries from compound
expressions. The wrapper is designed as numerical infrastructure: each floating
wrapper has exactly the size and alignment of its backend C object, the
default-state policy is explicit and predictable across threads and shared
libraries, and a small set of opt-in compile-time fast paths is available for
BLAS-like kernels. This manual documents the public API, the precision and
default-state model, the expression-template lifetime contract, and the
behavioral guarantees that callers can rely on.
\end{abstract}

\section{Introduction}

\textbf{gmpfrxx\_mkII} is a header-only C++17 wrapper library for GMP, MPFR, and
GNU MPC. In its default mode it requires no separate compiled library: including
a header and linking the relevant backend C libraries is sufficient. It exposes
four numeric domains.

\begin{itemize}[noitemsep]
  \item GMP-backed exact integers (\code{mpz_class}) and rationals (\code{mpq_class}).
  \item GMP-backed binary floating point (\code{mpf_class}) and an MPF-backed
        complex type (\code{mpfc_class}) built from two \code{mpf_class}
        components.
  \item MPFR-backed correctly-rounded real floating point (\code{mpfr_class}).
  \item GNU MPC-backed complex floating point (\code{mpc_class}).
\end{itemize}

\subsection{A from-scratch implementation}

This is a from-scratch implementation built directly on the GMP, MPFR, and MPC
\emph{C} APIs. It does not include \code{gmpxx.h}, does not link
\code{libgmpxx}, and does not bridge to or wrap any existing C++ wrapper. The
expression-template layer, the class designs, and the default-state machinery
are all original to this project. The result is compatibility-oriented in
spirit --- familiar type names and operator behavior --- but it is not a full
\code{gmpxx.h} clone, and it deliberately diverges where a clearer or more
predictable contract is worthwhile.

\subsection{Design priorities}

The library favors predictability over implicit convenience. The default
precision is a fixed, documented value; assignment to an existing object never
silently changes its precision; a single scalar constructor argument is always a
\emph{value}, never a precision selector; and the floating wrappers add no
per-object metadata around the backend C object, so arrays of wrappers keep
native stride. These choices target use as numerical infrastructure --- for
example dense linear-algebra kernels --- rather than ad hoc scripting
convenience.

\subsection{Public and internal namespaces}

The public API lives in two namespaces:

\begin{lstlisting}
namespace gmpxx;   // GMP-only API
namespace mpfrxx;  // MPFR API, plus the MPC API when the MPC header is included
\end{lstlisting}

The implementation lives in an internal namespace that is not part of the public
API:

\begin{lstlisting}
namespace gmpfrxx_mkII::detail;
\end{lstlisting}

In particular, \code{namespace gmpfrxx} is \emph{not} a public API namespace.
The \code{gmpfrxx_mkII} name appears only as the internal \code{detail}
namespace and as header and macro prefixes.

This manual assumes familiarity with the underlying C libraries. It describes
what the wrapper guarantees, not the internals of GMP, MPFR, or MPC.

GNU MPC & Required only for \code{mpfrxx::mpc_class}. \\
\bottomrule
\end{longtable}

\subsection{The MPFR TLS requirement}

The MPFR default precision, rounding mode, and exponent range are treated as the
single source of truth and are read from \code{libmpfr}'s thread-local state. The
project therefore requires an MPFR build with thread-local storage: a CMake
configuration must verify that \code{mpfr_buildopt_tls_p()} exists and reports
TLS enabled, and must fail otherwise. There is no fallback default-state
implementation for a non-TLS MPFR build.

\subsection{Compiling a program}

\begin{lstlisting}
#include <gmpfrxx_mkII.h>
#include <iostream>

int main() {
    mpfrxx::mpfr_class x = 2;
    mpfrxx::mpfr_class y = sqrt(x);   // 512-bit by default
    std::cout << y << '\n';
}
\end{lstlisting}

\begin{lstlisting}[language=bash]
c++ -std=c++17 example.cpp -lmpc -lmpfr -lgmp
\end{lstlisting}

If only real MPFR arithmetic is needed, include \code{mpfrxx_mkII.h} and link
\code{-lmpfr -lgmp}; GNU MPC is not required.

\section{Headers and namespaces}

Headers are split by required backend and link dependency so that a translation
unit pays only for the backends it uses.

\begin{longtable}{@{}llp{0.30\textwidth}@{}}
\toprule
\textbf{Header} & \textbf{Public API} & \textbf{Link dependency} \\
\midrule
\endhead
\code{gmpxx_mkII.h}   & \code{gmpxx::mpz_class}, \code{mpq_class}, \code{mpf_class}, \code{mpfc_class} & \code{-lgmp} \\
\code{mpfrxx_mkII.h}  & \code{mpfrxx::mpfr_class} (+ \code{mpz_class}, \code{mpq_class} aliases) & \code{-lmpfr -lgmp} \\
\code{mpcxx_mkII.h}   & \code{mpfrxx::mpc_class} & \code{-lmpc -lmpfr -lgmp} \\
\code{gmpfrxx_mkII.h} & aggregator of all of the above & \code{-lmpc -lmpfr -lgmp} \\
\bottomrule
\end{longtable}

\code{gmpxx_mkII.h} is deliberately GMP-only. It does not include \code{<mpfr.h>}
or \code{<mpc.h>}, references no \code{mpfr_*} or \code{mpc_*} symbol, and requires
neither library at link time. The GMP complex type \code{gmpxx::mpfc_class} is
implemented from two \code{mpf_class} components and stays in the GMP-only header
rather than being split into a separate complex header.

\code{mpfrxx_mkII.h} requires GMP and MPFR but not GNU MPC.
\code{mpcxx_mkII.h} includes \code{mpfrxx_mkII.h} automatically and layers the
complex type on top.

Two namespaces are used. \code{gmpxx} holds the GMP-only types.
\code{mpfrxx} holds the MPFR and MPC types and re-exports the GMP integer and
rational types as aliases:

\begin{lstlisting}
namespace mpfrxx {
    using ::gmpxx::mpz_class;
    using ::gmpxx::mpq_class;
}
\end{lstlisting}

\section{Public types overview}

\begin{longtable}{@{}llp{0.34\textwidth}@{}}
\toprule
\textbf{Type} & \textbf{Backend} & \textbf{Notes} \\
\midrule
\endhead
\code{gmpxx::mpz_class}   & GMP \code{mpz_t} & Exact integer. \\
\code{gmpxx::mpq_class}   & GMP \code{mpq_t} & Exact rational. \\
\code{gmpxx::mpf_class}   & GMP \code{mpf_t} & Binary float; no IEEE NaN/Inf/signed-zero. \\
\code{gmpxx::mpfc_class}  & two \code{mpf_class} & GMP complex; no GNU MPC dependency. \\
\code{mpfrxx::mpfr_class} & MPFR \code{mpfr_t} & Correctly-rounded real float. \\
\code{mpfrxx::mpc_class}  & GNU MPC \code{mpc_t} & Complex float. \\
\bottomrule
\end{longtable}

The two complex families are distinct by design. \code{gmpxx::mpfc_class} is
MPF-backed; \code{mpfrxx::mpc_class} is MPC-backed. Mixed arithmetic between the
families is intentionally rejected at compile time.

Because \code{mpf_class} and \code{mpfc_class} are GMP \code{mpf_t}-backed, they
do not model IEEE signed zero, NaN, or infinity, and cannot reproduce
signed-zero branch cuts. In particular \code{gmpxx::atan2(0,0)} returns $+0$ and
the four IEEE signed-zero cases are indistinguishable. Use \code{mpfr_class} or
\code{mpc_class} when signed-zero branch behavior is part of the numeric
contract.

\section{Quick start}

\begin{lstlisting}
#include <gmpfrxx_mkII.h>
using namespace gmpxx;
using namespace mpfrxx;

mpz_class n = 1;
for (int k = 1; k <= 50; ++k) n *= k;        // 50!

mpq_class q("22/7");
q.canonicalize();                            // arithmetic-ready

mpf_class a(1.0, 256), b = sqrt(mpf_class(2, 256));
mpf_class c = a + b * b;                     // evaluated immediately

mpfr_class x = const_pi();                   // 512-bit pi
mpfr_class y = sin(x / 6);

mpc_class z(mpfr_class(0), mpfr_class(1));    // i
mpc_class w = exp(z * const_pi());           // ~ -1
\end{lstlisting}

\section{Precision and default state}

\subsection{Initial defaults}

The wrapper default floating precision is \textbf{512 bits} for MPF, MPFR, and
MPC, unless overridden by environment variables (Section~\ref{sec:env}).

Assignment to an \emph{existing} floating object preserves the
left-hand-side precision in every domain. Changing a wrapper default affects only
\emph{subsequently constructed} objects.

\subsection{GMP MPF default precision}

GMP's \code{mpf_set_default_prec()} / \code{mpf_get_default_prec()} are
process-global, not thread-local. \code{mpf_class} therefore does not use GMP's
global default as its policy; it owns its own default. Default construction uses
\code{mpf_init2(..., gmpxx::default_mpf_precision_bits())}.

Two explicit escape hatches expose the GMP global C state for code that
intentionally wants it; these are \emph{not} the wrapper default policy:

\begin{lstlisting}
gmpxx::unsafe_get_gmp_global_mpf_precision_bits();
gmpxx::unsafe_set_gmp_global_mpf_precision_bits(bits);
\end{lstlisting}

\subsubsection{Default-context modes}

\code{GMPXX_MKII_DEFAULT_CONTEXT_MODE} is a program-wide build setting. All
translation units linked into one program must use the same value.

\paragraph{Frozen-env mode (default).}
Header-only; no provider object file; no header-owned mutable TLS. The default
MPF precision is 512 bits unless overridden by
\code{GMPXX_DEFAULT_MPF_PRECISION_BITS}, and the environment is read once and
then frozen. Runtime mutation of the wrapper default is unsupported:
\code{gmpxx::set_default_mpf_precision_bits(...)} is a compatibility no-op,
and \code{gmpxx::reload_default_mpf_precision_bits_from_environment()} keeps the
frozen value. Because the frozen value lives in a function-local static, each
linked image may observe its own frozen default; this is acceptable for the
simple header-only mode but is not a cross-DSO mechanism.

\paragraph{External-provider mode.}
Opt-in and not header-only: exactly one linked provider implementation owns the
mutable per-thread default-context storage. Missing provider symbols are a link
error. In this mode \code{gmpxx::set_default_mpf_precision_bits(...)} sets the
calling thread's provider-owned default and
\code{gmpxx::default_mpf_precision_guard} restores the previous value on scope
exit. Do not embed the provider object in multiple shared libraries; designate a
single owning library. Provider identity is checkable through
\code{gmpxx_mkII_default_context_provider_token_v1()}, and all participating DSOs
must observe the same token.

\subsection{MPFR default state}

MPFR default precision, rounding mode, and exponent range are owned by
\code{libmpfr}'s thread-local state; the wrapper keeps no separate mutable
storage for them. The public MPFR default API routes directly through MPFR:

\begin{lstlisting}
mpfrxx::default_precision_bits();          // mpfr_get_default_prec()
mpfrxx::set_default_precision_bits(bits);  // mpfr_set_default_prec()
mpfrxx::default_rounding_mode();
mpfrxx::set_default_rounding_mode(rnd);
mpfrxx::default_emin();  mpfrxx::default_emax();
mpfrxx::set_default_exponent_range(emin, emax);
\end{lstlisting}

\subsubsection{Per-thread first-use initialization}

On a thread's first wrapper default access, if the MPFR default state still has
its library-initial values (precision $=53$, rounding $=$ \code{MPFR_RNDN},
\code{emin}/\code{emax} at MPFR's defaults), the wrapper applies its 512-bit
policy together with any \code{MPFRXX_*} environment overrides. If application
code or another linked image has already changed any MPFR default field in that
thread, the wrapper leaves the existing state alone. A DSO-local one-shot flag
avoids repeatedly probing the defaults from hot paths; it stores no precision or
rounding value of its own.

To establish an explicit boundary --- for example before applying raw
\code{mpfr_set_default_prec(...)} --- call
\code{mpfrxx::initialize_thread_defaults()} first. After the one-shot
initialization has run, later raw MPFR default-state changes in the same thread
are not overwritten by implicit wrapper access.

Default setters affect only the calling thread, and only subsequently
constructed objects. \code{mpfrxx::reload_mpfr_defaults_from_environment()}
reloads environment-derived defaults for the calling thread only. Applications
that want consistent runtime defaults across worker threads must call a setter
or reload API inside each thread.

\subsection{MPC default state}

MPC inherits the MPFR thread-local defaults: without an MPC-specific override,
default real and imaginary precision equal the MPFR default precision, and
default real and imaginary rounding equal the MPFR default rounding.

MPC-specific overrides affect \emph{only} \code{mpc_class} defaults; they do not
touch MPFR defaults. Precision and rounding may be symmetric or asymmetric:

\begin{lstlisting}
mpfrxx::set_default_mpc_precision_bits(prec);              // symmetric
mpfrxx::set_default_mpc_precision_bits(real, imag);        // asymmetric
mpfrxx::set_default_mpc_rounding_mode(rnd);                // symmetric
mpfrxx::set_default_mpc_rounding_mode(real_rnd, imag_rnd); // asymmetric
\end{lstlisting}

Assigning a real value (an \code{mpfr_class}, \code{mpz_class}, \code{mpq_class},
or a supported scalar) to an \code{mpc_class} sets the imaginary component to
$+0$, following GNU MPC's \code{mpc_set_fr}-family semantics.

\subsection{Environment variables}\label{sec:env}

\begin{lstlisting}[language=bash]
GMPXX_DEFAULT_MPF_PRECISION_BITS

MPFRXX_DEFAULT_PRECISION_BITS
MPFRXX_DEFAULT_EMIN
MPFRXX_DEFAULT_EMAX
MPFRXX_DEFAULT_ROUNDING_MODE        # RNDN RNDZ RNDU RNDD RNDA RNDF

MPCXX_DEFAULT_PRECISION_BITS
MPCXX_DEFAULT_REAL_PRECISION_BITS
MPCXX_DEFAULT_IMAG_PRECISION_BITS
MPCXX_DEFAULT_ROUNDING_MODE
MPCXX_DEFAULT_REAL_ROUNDING_MODE
MPCXX_DEFAULT_IMAG_ROUNDING_MODE
\end{lstlisting}

Environment variables are process-startup configuration. Finalize all
\code{GMPXX_*}, \code{MPFRXX_*}, and \code{MPCXX_*} variables before first
wrapper default access and before worker threads begin numeric work. Concurrent
\code{setenv}/\code{unsetenv}/\code{putenv} racing with wrapper default reads is
outside the supported contract.

Invalid \emph{user} environment values do not terminate the process: they are
ignored with a diagnostic on \code{stderr} and the built-in default is used. By
contrast, an external default-context provider with a mismatched ABI version or
an internally inconsistent record may terminate the process after a diagnostic,
because that indicates a binary-integration error rather than a configuration
typo.

The \code{MPCXX_*} component variables override only their respective component;
the other component inherits the corresponding MPC default or, failing that, the
MPFR default. Asymmetric MPC environment precision and rounding are therefore
valid.

\section{Expression templates and the lifetime contract}

Public arithmetic operators (\code{+}, \code{-}, \code{*}, \code{/}, unary
\code{-}) and compound assignments return expression nodes rather than evaluated
values. A node may \textbf{borrow} an lvalue wrapper operand by raw reference and
may \textbf{own} an rvalue operand.

This is a user-visible contract: \textbf{do not bind an expression to
\code{auto} and let it outlive its operands.}

\begin{lstlisting}
// BAD: e holds raw references to a and b.
auto e = a + b;
mutate_or_destroy(a);
gmpxx::mpf_class r = e;          // dangling if a was destroyed,
                                 // or stale if a was mutated.

// GOOD: evaluate immediately.
gmpxx::mpf_class r = a + b;

// GOOD: explicit snapshot when a stable value is needed.
auto snapshot = gmpxx::mpf_class(a + b);
\end{lstlisting}

Expression node types are \code{[[nodiscard]]}, so a discarded standalone
expression such as \code{a + b;} produces a compiler diagnostic. This does not
make a saved node lifetime-safe.

\subsection{Generative random expressions}

Random getters follow the same policy as upstream \code{gmpxx.h}: they return
generator expression nodes, not value snapshots. This includes
\code{get_z_bits}, \code{get_z_range}, \code{get_f} on \code{gmpxx::gmp_randclass},
the matching getters on \code{mpfrxx::gmp_randclass}, and the MPFR floating
getters such as \code{get_fr(...)}. Evaluating such a node consumes the
associated \code{gmp_randstate_t} and may produce a fresh value.

The number of times a random node embedded in an expression tree is generated is
\emph{not} a semantic contract. If one draw must be reused, materialize it first:

\begin{lstlisting}
mpfrxx::mpfr_class r = state.get_fr(prec);
mpfrxx::mpfr_class y = r + r;     // reuses one draw
\end{lstlisting}

\code{gmp_randclass} generation is not internally synchronized. Use a state from
one thread at a time, or protect it with an external mutex.

\section{Scalar policy and common types}

Supported real scalar leaves are non-\code{bool} integral types up to 64 bits,
\code{float}, and \code{double}. Integral leaves are normalized to signed or
unsigned 64-bit storage before evaluation, so \code{long} and
\code{unsigned long} are accepted through that normalization rather than through
GMP's native \code{long} ABI. Supported scalars promote with \code{mpfr_class} to
\code{mpfr_class} and with \code{mpc_class} to \code{mpc_class}.

\code{bool}, \code{wchar_t}, \code{char16_t}, \code{char32_t}, \code{long double},
and \code{__int128} are intentionally outside the scalar-leaf contract and must
not become common-type promotion backdoors. (Direct \code{mpz_class}
construction from \code{__int128} is supported where the compiler provides it,
but \code{__int128} is still rejected as an expression scalar leaf.)

When \code{mpcxx_mkII.h} or \code{gmpfrxx_mkII.h} is included,
\code{std::complex<double>} is accepted as an MPC scalar operand. It promotes
with \code{mpc_class} to \code{mpc_class}, and with \code{mpfr_class} to
\code{mpc_class} because the result has a complex component. This support is
MPC-only and does not make \code{mpfrxx_mkII.h} depend on GNU MPC.

\section{Type reference}

This section lists the principal public surface of each type. Constructors,
accessors, and free functions follow GMP/MPFR/MPC naming conventions where a
direct correspondence exists.

\subsection{\code{gmpxx::mpz_class}}

\paragraph{Construction.} Default; copy/move; from \code{const char*} or
\code{std::string} with an optional base (base 0 auto-detects); explicit from
\code{double} (rejecting non-finite input) and from \code{mpz_srcptr}; implicit
from supported integral scalars (and from \code{__int128} where available);
explicit from \code{mpf_class} and \code{mpq_class}; from integer/rational
expression nodes. Construction from \code{bool} is deleted.

\paragraph{Number theory.}

\subsection{\code{gmpxx::mpq_class}}

\paragraph{Construction.} Default; copy/move; from string with optional base;

\paragraph{Math (\code{gmpxx} namespace).}
A self-contained MPF real-math set: \code{sqrt}, \code{sqr}, \code{abs},
\code{neg}, \code{ceil}, \code{floor}, \code{trunc}, \code{sgn}, \code{hypot},
\code{exp}, \code{log}, \code{log1p}, \code{expm1}, \code{sin}, \code{cos},
\code{tan}, \code{asin}, \code{acos}, \code{atan}, \code{atan2}, \code{sinh},

\subsection{\code{gmpxx::mpfc_class}}

A GMP-only complex type built from two \code{mpf_class} components.

\begin{lstlisting}
mpfc_class();
mpfc_class(const mpf_class& re, const mpf_class& im);
explicit mpfc_class(const mpf_class& re);             // im = 0
explicit mpfc_class(Scalar re);
mpfc_class(RealScalar re, ImagScalar im);
static mpfc_class with_precision(mp_bitcnt_t prec);
static mpfc_class with_precision(mp_bitcnt_t re_prec, mp_bitcnt_t im_prec);

\paragraph{Result precision of eager math.}
For a multi-operand eager math function, the result precision is chosen from the
MPFR object or MPFR-expression operands \emph{before} scalar and exact GMP
operands are materialized; those exact operands are then converted at the chosen
precision and do not implicitly promote the result to the current default
precision. If no MPFR operand contributes a positive precision, the wrapper
default precision is used.

\paragraph{Version-dependent functions.}
Some MPFR functions are backend-version features (for example \code{powr}, the
unit-based trigonometric functions, \code{rootn_si}, \code{compound_si},
\code{log2p1}, \code{exp2m1}, and intmax power helpers). The corresponding
wrappers are exposed only when the included \code{<mpfr.h>} advertises the entry
point; the rest of the MPFR wrapper remains available with older MPFR.

\paragraph{Comparisons.}
\code{mpfrxx::cmp(...)} is a total-order-style helper for ordered values; if
either operand is NaN it throws \code{std::domain_error}. The equality,
inequality, and ordering operators are NaN-aware predicates instead: equality is
false for unordered operands, inequality is true, and ordering predicates are
false when unordered.

\section{Rounding and evaluation contexts}

\subsection{MPFR rounding}

\code{mpfrxx::with_rounding(value, rnd)} returns a lightweight non-owning
target-bound handle that applies \code{rnd} to assignment and compound
assignment through that handle. It carries no precision field: the destination
\code{mpfr_class} determines the result precision exactly as a direct
\code{mpfr_*} call would. The handle must not outlive, be stored beyond, or be
used after destruction or move of the referenced object. It does not create a
dynamic rounding scope and does not affect independently materialized eager math
such as \code{sin(x)} or \code{const_pi(prec)}.

\code{mpfrxx::rounding_mode_scope} changes the calling thread's MPFR default
rounding mode for the lifetime of the scope, restores the previous value on
destruction, and updates the stable-rounding cache. Nested scopes are supported.
Eager MPFR math functions and constants observe the current default rounding mode
and therefore observe a \code{rounding_mode_scope}, but they are not affected by
target-bound \code{with_rounding} handles.

\subsection{MPC contexts}

MPC keeps a target-bound precision-and-rounding context because it needs separate
real and imaginary precision. \code{mpfrxx::mpc_evaluation_context} stores real
precision, imaginary precision, and an \code{mpc_rnd_t};
\code{mpfrxx::with_context(value, context)} returns a non-owning handle.
Assignment and compound assignment through the handle evaluate with the cached
rounding and target component precisions, avoiding per-operation default lookups
in fixed-context loops. The handle may also be built from separate real and
imaginary MPFR rounding modes, packed with \code{MPC_RND(real, imag)}. The handle
must not outlive the referenced object; normal builds check that the context
precision matches the target.

\section{Compile-time fast paths}

The three fast-path macros are compile-time contracts for hot kernels, enabled
deliberately per translation unit or target. They are independent: fixed
precision does not imply stable rounding, stable rounding does not imply fused
arithmetic, and fused arithmetic implies no fixed-precision contract.

\begin{longtable}{@{}p{0.32\textwidth}p{0.62\textwidth}@{}}
\toprule
\textbf{Macro} & \textbf{Meaning} \\
\midrule
\endhead
\code{GMPFRXX_MKII_FAST_FIXED_PREC} &
Applies to \code{mpf_class}, \code{mpfr_class}, \code{mpc_class}. Caller
guarantees that all participating floating objects in the hot path share one
fixed precision; a mismatch is a contract violation (asserted in debug builds).
Enables swap-based move assignment and bounded scratch reuse that skip
precision-restoration work. \\[0.4em]
\code{GMPFRXX_MKII_FAST_STABLE_RND} &
Applies to MPFR (and MPC paths inheriting MPFR rounding). Caller guarantees the
default rounding mode is stable during the kernel, except through wrapper setters
or \code{rounding_mode_scope}. Replaces repeated default-rounding reads with a
cached thread-local value. \\[0.4em]
\code{GMPFRXX_MKII_ENABLE_FMA} &
Applies to MPFR expression and compound-assignment paths. Maps \code{a += b*c}
to \code{mpfr_fma}, \code{a -= b*c} to \code{mpfr_fma} with an exactly negated
multiplicand, \code{a*b + c*d} to \code{mpfr_fmma}, and \code{a*b - c*d} to
\code{mpfr_fmms}. \\
\bottomrule
\end{longtable}

\paragraph{Numeric-semantics note.}
\code{GMPFRXX_MKII_ENABLE_FMA} \emph{changes} results: fused evaluation rounds
once where a materialized product-then-add rounds twice. This is why it is
separate from the fixed-precision macro, which does not change arithmetic
results. GMP MPF has no wrapper FMA: \code{a += b*c} on \code{mpf_class} may use a
direct multiply-then-add specialization but still follows MPF's two-step
rounding. Under \code{GMPFRXX_MKII_FAST_FIXED_PREC}, the MPF leaf-product
compound path may reuse a bounded header-local thread scratch object for the
intermediate product; this is a performance cache only and does not change MPF
precision semantics. If a scratch pool is exhausted by a deep expression tree,
the implementation falls back to a scoped backend temporary without changing
observable results, precision policy, or exception behavior.

\section{\code{mpq_class} stream input}\label{sec:mpqstream}

Stream extraction for \code{mpq_class} and raw \code{mpq_ptr} intentionally
matches upstream \code{gmpxx.h}. It uses \code{mpq_set_str}-compatible semantics,
preserves the parsed numerator and denominator, and does \emph{not} call
\code{mpq_canonicalize}. Input such as \code{2/4} remains \code{2/4} at the stream
layer, and structurally invalid raw rationals such as \code{1/0} or \code{0/0}
are accepted by extraction rather than setting \code{failbit}. The stream layer
is a raw serialization compatibility boundary, not an arithmetic-ready one.

Constructors, \code{set_str}, expression evaluation, and arithmetic-producing
paths continue to reject zero denominators and produce canonical rationals.
Before passing a stream-loaded value to GMP rational arithmetic, validate and
canonicalize it with \code{mpq_class::canonicalize()} (which first rejects a zero
denominator with \code{std::domain_error}, then calls \code{mpq_canonicalize}) or
\code{gmpxx::mpq_canonicalize_checked(mpq_ptr)}. If the canonicalization throws,
the target retains its original raw stream state and is not arithmetic-ready.

\section{Move semantics and layout guarantees}

Floating wrappers are nothrow move-constructible and add no per-object ownership
flag around the backend object. Their size and alignment are part of the
performance contract:

\begin{lstlisting}
sizeof(gmpxx::mpf_class)   == sizeof(mpf_t);
alignof(gmpxx::mpf_class)  == alignof(mpf_t);
sizeof(mpfrxx::mpfr_class) == sizeof(mpfr_t);
sizeof(mpfrxx::mpc_class)  == sizeof(mpc_t);
\end{lstlisting}

This preserves dense array stride for BLAS-like kernels: an array of
\code{mpfr_class} has the same stride as an array of \code{mpfr_t}, with no extra
padding from a wrapper-side validity flag.

Move construction keeps the moved-from source as a valid backend object by
initializing destination storage and swapping; this may allocate but avoids
ownership metadata. In normal builds, move assignment to a valid destination
preserves the destination precision. Under
\code{GMPFRXX_MKII_FAST_FIXED_PREC}, move assignment may assume that source and
destination precisions match and take the fast swap path; a mismatch is a caller
contract violation.

A moved-from floating object remains valid for destruction, assignment, and swap,
but its value and precision are unspecified until reassigned. Portable code must
not read a moved-from wrapper for numeric meaning or precision before assigning a
new value. \code{mpfc_class} follows MPF move behavior through its two
components.

\section{Platform notes}

\paragraph{Windows MPC division.}
On \code{_WIN32} builds, \code{mpc_class} division may use a scaled finite-value
workaround instead of calling \code{mpc_div} directly. It is a portability guard
for MinGW/MPC builds where a direct division can overflow near MPFR's
32-bit-style exponent limits even when the exact quotient is finite and small.
The workaround is narrow: it applies only when all components are finite, the
divisor is not $0+0i$, and at least one component's exponent is within a small
margin of the current MPFR \code{emin}/\code{emax}. It performs componentwise
max-scaling in MPFR temporaries and then rounds with the requested MPC rounding
mode. It is a compatibility path, not a separate complex-division semantics;
last-bit differences from a direct \code{mpc_div} are possible. Non-Windows
builds call \code{mpc_div} directly.

\paragraph{Interpretation.}
The committed data show three stable boundaries. First, reusable temporaries and
fixed-precision scratch paths control most GMP wrapper overhead in expression
benchmarks. Second, MPFR wrappers need loop-external rounding plus an
FMA-capable source shape before their hot loop reaches the raw C FMA call class.
Third, Rgemm is not primarily a wrapper problem in the current sweep: at
\code{N=1472}, the best GMP and MPFR wrappers are close to their raw C backend
class, while GMP remains about 1.3 times faster than MPFR in the best serial C
native kernels.

The benchmark reports should be consulted before drawing narrower conclusions.
They include static sorted tables, line plots, and representative disassembly
showing whether \code{mpf_init2}/\code{mpfr_init2}, rounding-mode lookup,
temporary construction, \code{mpfr_fma}, and OpenMP final reductions appear in
or outside the timed hot loop.
